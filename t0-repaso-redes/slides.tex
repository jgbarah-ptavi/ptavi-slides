\section{Conmutación}

\begin{frame}
\frametitle{Circuitos vs. paquetes}

\begin{itemize}
\item Conmutación de circuitos: Los equipos establecen un camino físico antes de la conexión que permanecerá ocupado durante la comunicación. Ejemplo: La Red Telefónica Conmutada.
\item Conmutación de paquetes: El emisor divide la información en paquetes, que llegan de manera independiente al destino. Ejemplo: Internet.
\end{itemize}

\end{frame}


\section{Principios de una red IP}

\begin{frame}
\frametitle{The End-to-End Principle (I)}

\begin{itemize}
\item Es una de las dos aproximaciones para diseñar un sistema de comunicación fiable a través de una red.
\item En una aproximación, el sistema puede pasar la responsabilidad para la entega correcta de los datos con los propios datos, asegurando fiabilidad en cada salto.
\item En la otra, la responsabilidad sobre los datos permanece en los puntos finales, asegurando fiabilidad punto a punto incluso si los puntos intermedios son no fiables.
\end{itemize}

\end{frame}


\begin{frame}
\frametitle{The End-to-End Principle (y II)}

\begin{itemize}
\item La consecuencia principal de este principio es que la inteligencia está cuanto más lejos de la red, mejor. Como los puntos intermedios carecen de responsabilidad, pueden ser simples y no necesitan ser robustos.
\item Así, tenemos un diseño que requiere puntos finales inteligentes y conscientes de la red y una red tonta.
\item Este diseño se sigue en Internet -probablemente la red más tonta que existe.
\item El sistema telefónico tradicional, por ejemplo, adopta el modelo contrario: una red inteligente y terminales tontos.
\end{itemize}

\end{frame}


\begin{frame}
\frametitle{TCP/IP y el modelo de referencia OSI}

\begin{itemize}
\item El modelo está compuesto por siete niveles
\item Cada nivel construye sobre servicios que provee el nivel inferior y ofreciendo servicios de abstracción al superior.
\end{itemize}

\end{center}

\end{frame}


\begin{frame}
\frametitle{TCP/IP y el modelo de referencia OSI (y II)}

\begin{itemize}

\item Para cada nivel del modelo, hay una comunicación (a nivel lógico) entre ese nivel y su equivalente en otro \emph{host}.

\item Podemos utilizar el modelo OSI para describir Internet (aunque no de manera perfecta):

\begin{itemize}
  \item La arquitectura de Internet ha evolucionado en el tiempo.
  \item En parte es anterior al modelo OSI.
  \item La estratificación no es tan estricta.
\end{itemize}

\item Por tanto, los protocolos \emph{reales} a veces tienen funcionalidad de varios niveles.

\end{itemize}

\end{frame}


\begin{frame}
\frametitle{El nivel de Red}

\begin{itemize}

\item En el nivel de red, un protocolo transforma un variedad de redes privadas en Internet: el Protocolo de Internet (IP).
\item El servicio que ofrece IP a los niveles superiores es sencillo: entrega de datagramas a un destino específico de la mejor manera posible (\emph{best effort}).
\item Es un servicio tan simple que puede ser utilizado por muchas implementaciones del nivel de enlace.
\item Pero la sencillez tiene un precio: IP no garantiza que la entrega llegue en un tiempo determinado; ni siquiera que se vaya a entregar.
\item Los paquetes IP se pueden perder, desordenar, retrasar o corromper en los niveles inferiores.
\end{itemize}

\end{frame}


\begin{frame}
\frametitle{El nivel de Red (y II)}

\begin{itemize}


\item IP no trata de corregir estos errores; le pasa el problema a los niveles superiores.
\item Pero provee de los siguientes servicios:
\begin{itemize}
  \item Fragmentación, en caso de que el datagrama sea más grande que la unidad máxima de transmisión de la nivel de enlace.
  \item Un campo de tiempo de vida (TTL) para prevenir \emph{loops}.
  \item Una etiqueta de tipo de servicio que permite dar prioridad a unos paquetes sobre otros.
  \item Un identificador del nivel superior para redirigir el paquete correctamente.
  \item Direccionamiento de los puntos finales (incluyendo la posibilidad de multicast).
\end{itemize}

\end{itemize}

\end{frame}

\begin{frame}
\frametitle{Más sobre IP}

\begin{itemize}

\item IP provee una abstracción de una única red, pero eso no cambia la naturaleza de las redes subyacentes.

\item Encima de IP, hay dos protocolos de transporte: Transmission Control Protocol (TCP) y User Datagram Protocol (UDP).

\item TCP se adapta al servicio IP para ofrecer fiabilidad, entrega ordenada, identificación de puertos y una velocidad de transmisión que se adapta a las características de la red.

\item UDP ofrece un servicio similar a IP, solamente añadiendo los puertos.

\item Los puertos son una abstracción que permiten tener varios servicios simultáneos.

\item Idea clave: IP provee abstracción, ocultando a la aplicación los detalles de la topología y los enlaces de la red, aislando los niveles inferiores de las necesidades de las aplicaciones.
\end{itemize}
\end{frame}

\item Los protocolos superiores pueden adaptarse o compensar el comportamiento de IP en cierto grado, pero bajos rendimientos a nivel IP tendrán como resultado bajos rendimientos de todo el sistema.
\end{itemize}

\section{Una nueva era}

\begin{frame}
\frametitle{Una nueva era}

\begin{itemize}
\item Internet está cambiando:
\begin{itemize}
  \item Los contenidos estáticos están dando paso a contenidos multimedia.
  \item El texto es reemplazado por música y audio.
  \item El audio y vídeo interactivo se están convirtiendo en algo del día a día.
\end{itemize}
\item Estos cambios requieren nuevas aplicaciones, y tienen asociados nuevos desafíos para los diseñadores de aplicaciones.

\item Los nuevos servicios necesarios son:
\begin{itemize}
  \item Gestión de la temporización.
  \item Detección y corrección de pérdidas.
  \item Identificación de la carga y de la fuente.
  \item Realimentación por parte del receptor sobre la calidad.
  \item Sincronización de elementos multimedia.
  \item Gestión de miembros.
\end{itemize}

\end{itemize}

\end{frame}


\begin{frame}
\frametitle{Nuevos protocolos}

\begin{itemize}
\item Los \emph{antiguos} protocolos no tienen capacidad para ofrecer estos servicios.
\item No hay un único protocolo que resuelva todas estas cuestiones.
\item Hace falta una combinación de protocolos.
\end{itemize}
